# ADR-0014: A host-agnostic core, and a localhost web host beside the desktop shell

## Status
**Accepted.** Extends — does not replace — **[ADR-0012](0012-tauri-shell-with-leptos-ui.md)** (the
Tauri shell + Leptos UI, and the IPC seam). The desktop app is unchanged in behaviour; this ADR is
about *where the code behind the seam lives* and adds a second host that reuses it.

## Context

GeleitMail is a Tauri desktop app: the Leptos/WASM UI (`geleit-ui`) reaches the engine only over the
Tauri IPC seam, whose commands live in `geleit-app` (ADR-0012). We now want a **web version** — the
same app usable from a browser — as the first step toward supporting more platforms.

A browser tab **cannot** run the mail engine: it has no API to open the raw TLS/TCP sockets IMAP (993)
and SMTP (465/587) require, and `geleit-engine` (async-imap, lettre, tokio sockets) does not even
compile to `wasm32`. So the engine must run in a native process regardless. For a **local-first,
privacy-first** product the only trust model that preserves the guarantee is **self-hosted**: the user
runs the server on their own machine, and their mail never leaves it. The browser then talks HTTP to
*their own* box instead of the Tauri webview talking IPC to the same process.

The obstacle was structural, not conceptual: every command's *logic* was welded to `tauri::State` and
lived in the `geleit-app` **binary** crate, so a second host could not call it without either dragging
Tauri (and WebKitGTK) into a headless server or duplicating subtle sync/offline-queue logic that would
inevitably drift.

## Decision

Split the code behind the seam into a **host-agnostic core** that every host adapts to, rather than a
Tauri-owned module a second host would have to copy.

1. **New crate `geleit-host`** holds `AppState` and the *logic* of all ~75 commands, plus the worker
   helpers and `message_html`. It depends on the engine crates and on **no host and no UI** — the same
   boundary `check-boundary.sh` guards for the engine crates (it is added to that check). Tauri does
   **not** enter its dependency tree, so a server built on it stays free of WebKitGTK.

2. **One injected seam, `trait Shell`.** The only things a command needs from its host that aren't data
   — *emit an event* to the frontend, *set the unread badge* — are injected through `Shell`, exactly
   the dependency-injection `AppState` already uses for `SecretStore`/`Notifier`. Native file dialogs
   are **not** in `Shell`: they already shell out to `zenity`/`kdialog` (a subprocess, not a Tauri
   API), so they are host-agnostic as-is. The auto-updater is **not** in `Shell` either — it is
   inherently Tauri (`tauri-plugin-updater`, ADR-0013), so it stays in the desktop host and the web
   host stubs it (a self-hosted server is updated by its operator).

3. **`geleit-app` becomes the thin Tauri adapter.** Each command is a one-line `#[tauri::command]`
   wrapper (generated by three small macros) that maps `tauri::State`/`AppHandle` onto the core's
   `&AppState`/`Shell`. `TauriShell` implements `Shell` by emitting over the webview bridge and setting
   the window title + tray tooltip. Behaviour is byte-for-byte what it was.

4. **New crate `geleit-server`** is the localhost web host (axum). It serves the *same* WASM UI bundle,
   and exposes the core over four routes: `POST /invoke/<cmd>` (the IPC seam, JSON in/out, via a
   dispatch `match` that is the analogue of Tauri's `generate_handler!`), `GET /mail/<id>` (a message's
   sanitized HTML on its own opaque origin with the **same** locked-down CSP as `mailproto`, S9.2), and
   `GET /events` (one Server-Sent-Events stream carrying the pushes `Shell::emit` produces). The UI is
   unchanged except one host-specific detail — the reading-pane iframe URL — which now goes through a
   host-provided JS shim (`geleitMailUrl`), the same npm-free shim pattern as `geleitInvoke`.

5. **Localhost-only, no auth (for now).** The server **binds `127.0.0.1`**, so nothing else on the
   machine can reach it and no login is needed on day one. It reuses the OS keychain and the same
   encrypted store as the desktop host, so a self-hosted GeleitMail keeps every byte of mail on the
   operator's own hardware. LAN access + TLS + a login token are a deliberate later, opt-in slice.

## Consequences

- **Positive:** two hosts, one tested body of mail logic — they cannot diverge. The desktop app is
  unaffected. The web version reuses the entire UI and engine; only the transport changed.
- **New dependencies (web host only):** `axum` (+`hyper`/`tower`/`tokio-stream`). They sit under
  `geleit-server`, never in the desktop binary, and go through `deny.toml`/supply-chain like anything
  else. The "no general HTTP *client*" posture (ADR-0012/0013) is untouched — this is an HTTP *server*.
- **Operational constraint:** SQLite is single-writer, so a given store is opened by **either** the
  desktop app **or** the server, not both at once. Documented, not yet enforced (a startup lock-check
  is a later slice).
- **Background workers now run in both hosts (done):** the scheduler, per-account IMAP IDLE watchers,
  and the full-mailbox backfill moved into `geleit-host::worker`, each a host-agnostic `run(state, …)`
  loop the host spawns on its own runtime (desktop via `tauri::async_runtime::spawn`, web via
  `tokio::spawn`). Only the scheduler needs a `Shell` (to emit `mail-arrived` + set the badge); IDLE and
  backfill need only `AppState`. The web host therefore auto-syncs and pushes new mail over SSE with no
  manual Refresh — live-verified against local Dovecot.
- **Web-native file I/O — the common flows are done:** `save_eml`, `save_attachment`, and `pick_files`
  (compose attachments) go through the browser on the web host — downloads via `GET /download/…` with
  `Content-Disposition`, uploads via `POST /upload` into a staged temp path `send_message` reads. The
  reusable byte-generation (`eml_bytes`/`attachment_bytes`/`stage_upload`) lives in `geleit-host`, so
  the desktop's native-dialog path and the web's HTTP path share one implementation. `api.rs` picks the
  path via a `geleitIsWeb` shim, so the UI call sites are unchanged. Still deferred: **folder/account
  *export*** on the web (they return a summary that doesn't map to a single browser download — a `.zip`
  is the likely answer; they keep working via `zenity` on localhost meanwhile), and `open_eml_file`.
- **Deferred, tracked (not silently dropped):**
  - Folder/account export as a web download (see above); a staged uploads GC (sent messages' temp files).
  - A strict app-document CSP is set as an HTTP header (`script-src 'self' 'wasm-unsafe-eval'`, etc.);
    tightening it to match the Tauri config exactly is polish.

## Alternatives rejected

- **Engine-in-the-browser + a WebSocket↔TCP bridge** ("fully client-side"): technically possible (TLS
  inside the browser over the WS, so the bridge sees only ciphertext) but a months-long WASM port of
  async-imap/lettre/tokio/TLS **and** a regression — background sync (IDLE, backfill, new-mail push)
  dies when the tab closes. Rejected: much more work for a worse always-on-mail story.
- **Server links `geleit-app` as a lib:** drags Tauri + WebKitGTK into a headless server. Rejected.
- **Server reimplements the commands against the engine directly:** duplicates subtle sync/offline-queue
  logic that would drift. Rejected (this ADR exists to avoid exactly that).
