# S0.2 — Sandboxed HTML email rendering spike · Findings

Outcome of the M0 slice S0.2 feasibility spike (`specs/m0/s0.2-html-spike/`). Feeds ADR-0001
and the S0.4 UI decision. Spike code is throwaway (`spikes/s0.2-html-render/`, excluded from
the workspace).

## Question
Can a **sandboxed webview component** render real-world HTML email **without leaking to the
network** and **without executing hostile code** — on a native (Slint) app, with a credible
cross-platform path? (ADR-0001.)

## Setup
- **Webview:** `wry` 0.55 over WebKitGTK 2.52 (Linux). Window via `tao` 0.35.
- **Safety mechanism tested:** pre-render HTML **sanitization** with `ammonia` 4 (drops
  `<script>` and `on*` handlers; URL schemes restricted to `cid`/`data`/`mailto` so remote refs
  are dropped; `link`/`style`/`iframe`/`object`/`embed`/`base` removed).
- **Adversarial fixture:** every vector targets a **distinct** `203.0.113.x` host (TEST-NET-3,
  RFC 5737 — non-routable, safe, literal IPs so each `connect()` is attributable and DNS-free):
  `.5` tracking pixel · `.6` remote image · `.7` script-created image · `.8` onerror · `.10`
  remote CSS · `.11` font · `.12` CSS background.
- **Two oracles:** (1) `strace -f -e trace=connect` → which hosts got a real outbound
  `connect()`; (2) a `wry` **IPC oracle** — a wry-injected init script proves the JS *engine*
  is on, and the email's own inline `<script>` posts `PAGE-SCRIPT-RAN` iff it executes.
  (Network-independent, so it can't be confounded by lazy loads or coalescing.)

## Results (evidence in `spikes/s0.2-html-render/evidence/`)

| Run | remote `connect()` hosts | non-loopback connects | JS engine | email's `<script>` ran |
|---|---|---|---|---|
| adversarial **raw** | `.5`, `.6`, `.10` | **3** | on | **no** |
| adversarial **sanitized** | none | **0** | on | no |

- **Privacy gate — PASS.** Sanitized rendering made **zero** outbound connections (verified: no
  `AF_INET` connect of any kind in `adversarial-sanitized.strace`). Raw leaked to three hosts
  (tracking pixel, remote image, remote stylesheet — all eagerly loaded), proving the test is
  real. (`adversarial-raw.strace` vs `adversarial-sanitized.strace`.)
- **JS / no-script — nuanced, reported honestly:**
  - The JS **engine is on** (our injected `init-script-ran` IPC fires in both runs) — wry/WebKitGTK
    runs JavaScript by default and exposes no builder toggle to disable it.
  - The **email's own inline `<script>` did not execute** in `wry`'s `with_html` mode — even in
    the **raw** run, `PAGE-SCRIPT-RAN` never fired and the script-created image (`.7`) never
    connected. This appears favorable, but it is **undocumented behavior we will not rely on.**
  - After **sanitization** the `<script>` is gone entirely (`contains_script=false`,
    1417→~300 bytes), so it cannot run regardless.
- **Rendering — OK.** `newsletter`, `receipt`, `multipart` rendered and exited cleanly
  (`*.render.stderr`); `adversarial` rendered under strace. Screenshots intentionally omitted
  (would capture the live desktop); render-without-error + the sanitized HTML dump are the evidence.

## Key caveat — sanitization vs. fidelity
`ammonia`'s default policy **also strips inline `style` attributes** (see
`evidence/newsletter.sanitized.html`: structure kept, all CSS lost). Safe, but visually poor for
real HTML email, which leans heavily on inline CSS. **M3/M4 must build a CSS-aware sanitizer**
that keeps safe inline CSS while stripping remote `url(...)`, `@import`, and dangerous properties.

## Cross-platform path (AC4)
`wry` is the same component Tauri uses across desktop OSes: **WebKitGTK** (Linux, tested here),
**WKWebView** (macOS), **WebView2** (Windows). Sanitize-before-render is engine-independent, so
the guarantee carries across platforms. Unknowns to validate later: per-OS JS-disable
(WebKitGTK has a settings toggle; verify WKWebView/WebView2 equivalents), and webview-layer
remote-content blocking as defense-in-depth.

## Recommendation (feeds ADR-0001 / S0.4)
**Confirm ADR-0001:** a sandboxed webview component (wry/WebKitGTK) can render HTML email
safely. The privacy gate is proven; safety rests on **pre-render sanitization** (zero network
leakage, scripts removed). Carry forward to M3/M4:
1. Build a **CSS-aware sanitizer** (fidelity without remote leakage).
2. **Explicitly disable JavaScript** in the webview (WebKitGTK settings; verify per-OS) — do
   not rely on the observed non-execution of page scripts.
3. Add **webview-layer remote-content blocking** as defense-in-depth and the basis for the
   per-message "load remote content" opt-in (PRIV-2).

Final commit to Slint still depends on S0.3 (virtualized list) per ADR-0001.
