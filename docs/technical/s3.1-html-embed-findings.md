Spike: `spikes/s3.1-html-embed` (throwaway). Question: can a `wry` webview be embedded **inside the
Slint window** to render email HTML, sandboxed â€” realising ADR-0001 (which had only been validated

## Update (post-S4): Slint must render in **software**, not GL.

Slint's default GL renderer (femtovg) and the webview's GL (webkit) do **not** coexist on at least
some X11 setups â€” they produce an async `GLXBadWindow` that crashes the window (it bit on eager
startup build; see PR #52/#53). Fix: the app forces `SLINT_BACKEND=winit-software` (feature
`renderer-software`) so Slint has **no GL context** to collide with. winit window handle access
(`with_winit_window`) + `build_as_child` are unaffected by the renderer choice.

## Result: yes, on X11.

- **Embedding works.** `slint` (feature `unstable-winit-030`) exposes the winit window via
  `WinitWindowAccessor::with_winit_window`; `winit::Window` implements `raw-window-handle 0.6`'s
  `HasWindowHandle`, which `wry 0.55`'s `WebViewBuilder::build_as_child` accepts. The webview is a
  child window positioned with `with_bounds` / `set_bounds`, shown/hidden with `set_visible`.
- **GTK loop must be pumped.** webkit2gtk needs GTK initialised (`gtk::init()`) and its event loop
  iterated; under Slint's event loop we pump it from a repeated `slint::Timer`
  (`while gtk::events_pending() { gtk::main_iteration_do(false) }`). With this, the webview renders.
- **Platform scope:** `build_as_child` is **X11 only** (wry docs). On Wayland the gtk-container path
  (`WebViewBuilderExtUnix::new_gtk`) is needed â€” a follow-up. Our integration degrades gracefully:
  if `build_as_child` is unavailable, it's a no-op and the plain-text reading pane stays.

## Security (PRIV-1 / PRIV-4)

- **Sanitization is the guarantee.** `ammonia` (engine `safehtml::sanitize_html`, allowed schemes
  `mailto`/`cid` only) strips `<script>`, `on*` handlers, and **every `http(s)` reference** (remote
  images/trackers, remote links) â€” confirmed by the spike output and unit tests.
- **The webview *does* execute page JS** (a script injected *after* sanitization fired). So
  sanitization is load-bearing; real email HTML always passes through it (scripts removed â†’ none
  run). **S3.2** adds defense-in-depth: a strict `Content-Security-Policy` (`script-src 'none';
  img-src 'none'` â€¦) so even a sanitizer miss can't load/run anything.
