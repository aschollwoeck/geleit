# Backlog — Fix HTML crash #2 (winit GLXBadWindow panic on small windows)

Reported: clicking a mail message crashes again (black loading screen → crash).

## Root cause (reproduced)
A Rust **panic** in winit: `set_theme` → `flush_requests().expect("Failed to change window theme")`
surfaces a stray async **`GLXBadWindow`** generated by the embedded webkit (GL) webview. The software
renderer fixed Slint's GL, but the webview still uses GL, and on X11 a small/cramped webview surface
intermittently produces this error. Correlates with **small windows**: a 500px window crashes;
≥700px and maximized are stable. (Distinct from the earlier degenerate-size `BadValue`, PR #91.)

## Mitigation
Only build/show the embedded webview when the window + body are comfortably large
(`win ≥ 800×540`, body ≥ 360×240); otherwise fall back to the sanitized **text** view. This removes
the reproducible triggers (small windows, narrow reading pane from a wide-dragged list) — HTML in a
cramped pane wasn't useful anyway. Also refreshed the stale HTML-only fallback text.

## Acceptance criteria
1. build/test/clippy -D warnings (+dangerous-tls)/fmt/`cargo deny check` green.
2. The reproduced crash (500px / 700px window, open HTML) no longer crashes; HTML still renders when
   maximized. (Verified: old binary panics with GLXBadWindow; new binary survives tiny/narrow/maximized
   and renders HTML when maximized.)

## Honest limitation
The embedded webview is inherently fragile on X11 (async GL errors). This removes the reproducible
small-window triggers but can't guarantee zero flaky crashes on large windows without deeper work
(or a non-webview HTML path). If crashes persist on a large window, capture the terminal output.

## Deliverables
- `body_too_small` gates on window + body size; updated fallback text.
